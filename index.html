Educational Content & Overviews

CORE IDEA

Avoid comparisons by treating each value as a memory address (index). Use a frequency table (Count Array) to map numbers directly to their final sorted destination in linear time.

HOW IT WORKS (STEP BY STEP)

  1. Find the Maximum Value (k) in the input array to define the range.
  2. Create a Count Array of size k+1 and increment index Count[value] for every element found.
  3. Transform the Count Array into Prefix Sums (accumulate values) to find the exact ending position for each number.
  4. Iterate backward through the original array and place each element into the Output Array according to its index in the Prefix Count.

COMPLEXITY

Worst / Average: O(n + k) | Space: O(n + k) | Stable: Yes

WHEN TO USE

Best for sorting non-negative integers when the range of values (k) is not significantly larger than the number of elements (n). Extremely efficient for sorting grades, ages, or small IDs.

CORE IDEA

Sort multi-digit numbers by processing individual digits. By applying a stable sub-sort (such as Counting Sort) to each digit position, the array becomes fully sorted without comparing entire numbers directly.

HOW IT WORKS

  1. Find the maximum value to determine the number of digits (d).
  2. Start with the least significant digit (LSD).
  3. Perform a stable Counting Sort on that digit using buckets based on the chosen base (e.g., base-10 → 0–9).
  4. Move to the next digit (tens, hundreds, etc.).
  5. Repeat until the most significant digit is processed.

Stability is essential to preserve ordering from previous passes.

COMPLEXITY

  • Time: O(d(n + k))
  • Space: O(n + k)
  • Stable: Yes
Where: d = number of digits, k = base (e.g., 10), not the full value range

WHEN TO USE

  • Efficient for large datasets of integers or fixed-length strings.
  • Best when: n is large and d is small.
  • Preferred over Counting Sort when the value range is huge.

NOTES

  • Works best with non-negative integers.
  • Can be adapted for negatives and floats with preprocessing.
  • Alternative variant: MSD Radix Sort (left → right).

What it does: Counting Sort and Radix Sort are non-comparison-based sorting algorithms that sort elements by distributing them based on their integer values or specific digit representations, rather than comparing elements directly against each other.

How it works:
- Counting Sort: Counts the frequency of each distinct element in the input array. It uses an auxiliary "count" array where the index represents the element value and the stored number is the frequency. It then computes prefix sums to determine the absolute sorted position of each element.
- Radix Sort: Processes numbers digit by digit, from least significant to most significant (LSD). For each digit pass, it uses a stable sorting algorithm (like Counting Sort) to group the numbers into standard base-10 buckets (0-9). By the end of the most significant digit pass, the array is fully sorted.

When to use it:
- Use Counting Sort when the range of potential values (k) in the input is relatively small compared to the number of elements (n). It is extremely fast (linear time O(n+k)) when k = O(n).
- Use Radix Sort for sorting lists of integers, strings, or records with multiple keys when $n$ is large and the maximum number of digits ($d$) is relatively small. It avoids the massive memory overhead Counting Sort would suffer on large integer ranges.

Let's manually trace Counting Sort on the array: Input = [4, 2, 2, 8, 3, 3, 1]

  1. Find Max: The maximum value is k = 8. We need a Count array of size 9 (indices 0 to 8).
  2. Count Frequencies: We count how many times each number appears.
    Count = [0, 1, 2, 2, 1, 0, 0, 0, 1] (e.g. index 2 is 2 because the number '2' appears twice).
  3. Compute Prefix Sums: We modify the Count array such that each element at each index stores the sum of previous counts. This tells us the final ending positional index for each number.
    Prefix Count = [0, 1, 3, 5, 6, 6, 6, 6, 7]
  4. Place in Output (Iterating backward for stability):
    - Look at the last element in Input: 1. Check Prefix Count[1], which is 1. We place 1 at Output index 1-1 = 0. We decrement Prefix Count[1] to 0.
    - Look at next element from the right: 3. Check Prefix Count[3], which is 5. We place 3 at Output index 5-1 = 4. We decrement Prefix Count[3] to 4.
    - Look at next element: 3. Check Prefix Count[3], which is now 4. We place 3 at Output index 4-1 = 3.
    - This process continues until all elements are placed exactly into their mapped topological indices!
    Final Output = [1, 2, 2, 3, 3, 4, 8]
Important Ideas
  • Integer Exploitation: These algorithms don't ask "Is A > B?". Instead they ask "What IS A? Ah, A is 5, so put it in bucket 5." They use the values themselves as direct memory addresses/indices.
  • Stability is Mandatory: The Counting Sort used *inside* Radix Sort MUST be stable. A stable sort preserves the relative order of identical elements. If it wasn't stable, sorting the "Tens" digit would scramble the work you just finished on the "Units" digit!
Common Mistakes
  • Mistake: Using Counting Sort on floating-point numbers or negative numbers out-of-the-box.
    Correction: Standard Counting sort maps values directly to list indices, which must be non-negative integers. Negative numbers require an offset shift.
  • Mistake: Using Counting Sort when the scale k is 10^9.
    Correction: Trying to sort the array [1, 1000000000] would painfully generate an array of one billion empty zeroes just to sort two elements, crashing the browser's memory limits!
  • Mistake: Assuming Radix sort always beats O(n log n) QuickSort in the real world.
    Correction: Radix sort has a high constant factor overhead for memory allocation and bucket copying. For general-purpose arrays, heavily optimized QuickSorts often win empirically due to hardware cache locality!

Ω(n log n) Lower Bound (Decision Tree)

Comparison sorts must evaluate leaves in a binary decision tree. For n items, there are n! permutations (leaves). The tree height must logically be at least log(n!), which simplifies asymptotically to Omega(n log n).

Observe how QuickSort incrementally partitions the array and physically Swaps elements based on < / > Comparisons. Radix/Counting Sort, instead, relies purely on Array Reads/Writes to map elements to frequency auxiliary arrays and reconstruct them sequentially.

Role of the Sorting Subroutine (Impact): Notice inside the Radix Sort visualization that it iteratively runs Counting Sort on each digit position (Units, Tens, Hundreds). If this underlying inner-sort was NOT STABLE (meaning it swapped equal elements arbitrarily), the work done during previous, less significant digit passes would be completely destroyed, rendering the final array incorrectly sorted! You can see stability preserved as elements iterate sequentially from right-to-left into their sorted positions based strictly on the current digit slice without overlapping earlier orderings.

Counting and Radix sorts do not natively follow standard Divide and Conquer (D&C), Dynamic Programming (DP), or Greedy paradigms. They are completely dependent on integer properties.
